# Project Adaptation

## How To Adapt To Any Project

Extract these before writing:

- product name and category
- target buyer and end user
- what the buyer cares about economically or operationally
- current proof points and what cannot be claimed
- forbidden terms, legal/compliance limits, medical or financial sensitivities
- default next sales step
- allowed links and when to use them
- prior call transcripts or customer language

If a project has an `AGENTS.md`, `.agents/product-marketing-context.md`, sales notes, or call transcripts, prefer that language over generic outbound formulas.

## Proof Point Handling

Ask which proof points are allowed now. If the project contains metrics, offer them as options and classify them:

- safe external proof: customers, named case studies, public reviews, public benchmarks
- safe internal proof: product telemetry, usage metrics, pilot data, anonymized aggregates
- sensitive proof: revenue, retention, health/clinical claims, private customer names
- not allowed: unverified claims, guaranteed outcomes, misleading averages

Use one proof point at a time. If proof is weak, use a credible product mechanism instead of inflating results.

## Banned Terms And Claims

Always ask whether there are banned words or claims. If project context suggests them, propose a prefilled list.

Examples:

- channel confusion: "бот", "Telegram-ассистент", "Telegram-приложение" when the product is cross-platform
- medical overclaiming: "лечит", "диагностирует", "терапия", "замена психолога"
- vague HR language: "ментальное здоровье" when the buyer needs operational language
- weak endings: "если актуально", "давайте обсудим", "напишите, если интересно"
- AI tells: repeated `—`, over-polished abstractions, generic personalization

## Buyer Lenses

### HRD / People Director

Care about: adoption, employee trust, prevention, engagement, confidentiality, manager load, measurable rollout.

Avoid: making it sound like an extra HR project with manual work.

### CEO / Founder / Team Owner

Care about: execution speed, leadership bandwidth, retention, delivery quality, morale before it becomes turnover, cost of overload.

Avoid: HR jargon and soft language that does not connect to business consequences.

### Procurement / Finance

Care about: risk, implementation effort, cost, measurable criteria, pilot scope, vendor clarity.

Avoid: emotional argument without measurable pilot logic.

## PRACTIM-Specific Guardrails

Use these only when the current project is PRACTIM or the user asks for PRACTIM copy.

- Say "продукт PRACTIM" or "приложение".
- Do not say "Telegram-ассистент", "Telegram-приложение", or "бот" as product positioning.
- Position as workplace self-help and employee wellbeing, not therapy or medical diagnosis.
- Use 3-month pilot as the default pilot frame unless the user changes it.
- Mention privacy carefully: company sees only aggregated/anonymized usage, not personal dialogs.
- Useful topic angles: stress, anxiety, burnout, procrastination, difficult conversations, focus loss, recovery after peak load.
- If using product telemetry, treat it as product usage data, not clinical proof.

## Link Use For PRACTIM

If the user wants practice links, ask which topic and which link:

- burnout / выгорание
- anxiety / тревожность
- impostor syndrome / синдром самозванца
- procrastination / прокрастинация
- another topic

Place the link near the end as a secondary way to feel the product, not as the main pitch.
