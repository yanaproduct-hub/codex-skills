# Project Adaptation

## Adapt To Any Project

Extract these before writing:

- product name and category
- target buyer and end user
- what the buyer cares about economically or operationally
- current proof points and what cannot be claimed
- forbidden terms, legal/compliance limits, medical or financial sensitivities
- default next sales step
- allowed links and when to use them
- prior call transcripts or customer language

If a project has `AGENTS.md`, `.agents/product-marketing-context.md`, sales notes, or call transcripts, prefer that language over generic outbound formulas.

## Proof Point Handling

Ask which proof points are allowed. If project metrics exist, offer them as options and classify them:

- safe external proof: customers, named case studies, public reviews, public benchmarks
- safe internal proof: product telemetry, usage metrics, pilot data, anonymized aggregates
- sensitive proof: revenue, retention, health/clinical claims, private customer names
- not allowed: unverified claims, guaranteed outcomes, misleading averages

Use one proof point at a time. If proof is weak, use a credible product mechanism instead of inflating results.

## Buyer Lenses

### HR / People

Care about: adoption, employee trust, prevention, engagement, confidentiality, manager load, measurable rollout.

Avoid making the product sound like another manual People initiative.

### CEO / Founder

Care about: execution speed, leadership bandwidth, retention, delivery quality, morale before it turns into turnover, cost of overload.

Avoid soft HR language that does not connect to business consequences.

### Technical Buyer

Care about: security, implementation effort, data handling, integration path, reliability, admin load.

Avoid vague emotional framing or overclaiming the behavioral outcome.

### Procurement / Finance

Care about: risk, implementation effort, cost, measurable criteria, pilot scope, vendor clarity.

Avoid an emotional argument without measurable pilot logic.

## English Market Nuance

For US/UK-style cold outbound:

- be brief and specific
- do not over-apologize
- do not sound overly formal
- do not over-flatter
- use one simple CTA
- use "worth exploring?", "worth a look?", "should I send this to someone else?" carefully and only when the line fits the context
- for executives, write shorter and more directly
- for HR/People, balance employee language with business impact

## Banned Terms And Claims

Always ask whether there are banned words or claims. If project context suggests them, propose a prefilled list.

Examples:

- medical overclaiming: "treats", "diagnoses", "therapy", "clinical proof"
- fake certainty: "guaranteed", "proven to eliminate", "will increase revenue"
- sales cliches: "leverage", "synergy", "best-in-class", "game-changing"
- weak endings: "let me know if you're interested"
- AI tells: over-polished abstractions, generic personalization, repetitive sentence rhythm
